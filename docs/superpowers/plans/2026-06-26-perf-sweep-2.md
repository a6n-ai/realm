# Perf Sweep 2 — Missing Indexes, Bundle Split, Render & Query Waterfalls Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Land the confirmed perf findings from the second review pass — add the four missing FK/activity indexes that today force full table scans on the hottest staff pages (pipeline list, customer 360, order detail), split the heavy `PhoneInput` flag bundle out of the three sheet-gated dashboard forms, memoize the orders-list status counts, and collapse a handful of independent-await waterfalls and a two-query-to-one-join on the customer meals page. No behavior changes — pure latency/bytes wins.

**Architecture:** Four independent tracks. (1) **Indexes**: one `db:generate` migration adds composite indexes on `inquiry_activities(inquiry_id, created_at)`, `orders(user_id, created_at)`, `payments(order_id)`, and `order_activities(order_id, created_at)` — each chosen so a single index serves both the FK filter and the `created_at desc` sort/limit the callers do. (2) **Bundle**: `next/dynamic { ssr: false }` the `PhoneInput` in the three dashboard sheets, matching the pattern already shipped on the public signup/contact forms. (3) **Client render**: derive all `OrdersList` status-pill counts in one memoized pass instead of 6+1 full-array filters per keystroke. (4) **Query shape**: fold independent reads into `Promise.all` / a single join (inquiry detail, meals page, `inquiriesService.create`, `pickDish`). A fifth, lowest-priority track addresses the `lower()`-wrapped users dedup lookup.

**Tech Stack:** Next.js 16 (non-stock — read `node_modules/next/dist/docs/` before any framework change), Drizzle ORM + drizzle-kit 0.31, React 19, Vitest, live-DB Postgres harness.

### Task 1: Add the four missing indexes (one migration)

**Files:**

- Modify: `apps/web/db/schema/orders.ts`
- Generate: `apps/web/db/migrations/<next>_*.sql` (+ snapshot/journal) via `db:generate`

**Findings covered:** `inquiry_activities.inquiry_id` unindexed (inquiries.ts:47-56), `orders.user_id` unindexed (orders.ts:12-38), `payments.order_id` unindexed (orders.ts:40-45), `order_activities.order_id` unindexed (orders.ts:47-54).

Highest impact-to-effort in the sweep. Today these tables carry only `baseColumns`/`updatableColumns` (id/public_id/created_at + audit), so every FK lookup the staff pages do is a sequential scan:

- `inquiriesService.listForPipeline` (inquiries.service.ts L242-289) hits `inquiry_activities` three ways with zero index support — the `agg` CTE `GROUP BY inquiry_id` (L248), a **per-row correlated subquery** `select next_follow_up_at … where inquiry_id = inquiries.id order by created_at desc limit 1` (L275-279, a full scan per pipeline row → super-linear as activity volume grows), and `listActivities` filter+sort (L294-298).
- `getCustomer360` (customers.service.ts L98-112) scans+sorts the whole `orders` table on `where user_id = ? order by created_at desc` — the core of every customer detail view.
- `readOrder` (orders.service.ts L244-247) scans `payments` on `order_id = ?` per order-detail load.
- `listOrderActivities` (orders.service.ts L251-252) scans+sorts `order_activities` on `order_id = ? order by created_at desc`.

Composite `(fk, created_at)` indexes are chosen so one index serves both the filter and the `created_at desc` ordering / `limit 1`.

- [ ] **Step 1: Index `inquiry_activities`**

`inquiries.ts` already imports `index`. Add the missing third table-config arg to `inquiryActivities`:

```ts
// before — no index block
export const inquiryActivities = pgTable("inquiry_activities", {
  ...baseColumns("iac"),
  inquiryId: bigint("inquiry_id", { mode: "bigint" }).notNull().references(() => inquiries.id, { onDelete: "cascade" }),
  // ...
});

// after — add (t) => [...]
export const inquiryActivities = pgTable("inquiry_activities", {
  ...baseColumns("iac"),
  inquiryId: bigint("inquiry_id", { mode: "bigint" }).notNull().references(() => inquiries.id, { onDelete: "cascade" }),
  // ...
}, (t) => [
  index("inquiry_activities_inquiry_created_idx").on(t.inquiryId, t.createdAt.desc()),
]);
```

This single index serves the correlated latest-touch subquery (`inquiry_id` eq + `created_at` desc limit-1), `listActivities`, and the `GROUP BY inquiry_id` aggregate.

```tsx
const counts = useMemo(() => {
  const c: Record<string, number> = { all: rows.length };

// memoize filtered (was L46):
const filtered = useMemo(
```

- [ ] **Step 2: Meals page — two queries → one join**

The first query (L18-22) fetches only `users.id` to translate the session publicId so the second (L26-47) can filter `orders.userId` — two serial round trips on the customer's primary page. Replace with one `orders innerJoin deliveryFrequencies innerJoin users` filtered by `users.publicId`:

```tsx
const [orderRow] = await db
  .select({
    id: orders.id,
    publicId: orders.publicId,
    userId: orders.userId,
    planId: orders.planId,
    persons: orders.persons,
    mealSlots: orders.mealSlots,

    includeSunday: orders.includeSunday,

    pausedUntil: orders.pausedUntil,

  .innerJoin(users, eq(orders.userId, users.id))
```

### Task 5: Make the users dedup lookup index-able (lowest priority)

**Files:**
- Modify: `apps/web/lib/services/customers.service.ts`
- Optionally modify: `apps/web/db/schema/auth.ts`

**Findings covered:** `findExistingByContact` filters `lower(phone)`/`lower(email)` but the users unique indexes are on the raw columns (customers.service.ts:75-84). Severity low, effort M — explicitly the lowest-value apply-now item; the reviewer notes it is "low priority while the users table is small."

`findExistingByContact` (L76-77) queries `where lower(users.phone)=? OR lower(users.email)=?`, but the only users indexes are `uniqueIndex` on the **raw** `phone`/`email` columns (auth.ts L27-28, partial `where … is not null`). The `lower()` wrapper prevents those indexes from being used, so this dedup lookup sequentially scans users. Two options — prefer **A** (no migration, reuses existing unique indexes):

- [ ] **Option A (preferred): match on raw equality**

Phones are stored canonical E.164 and emails are validated/normalized at write time, so the `lower()` wrapper is redundant for the stored data. Match the raw columns so the existing partial unique indexes serve the predicate:

```ts
// before
const conds = [eq(sql`lower(${users.phone})`, phone.toLowerCase())];
if (email) conds.push(eq(sql`lower(${users.email})`, email.toLowerCase()));

// after — reuses users_phone_unique / users_email_unique
const conds = [eq(users.phone, phone)];
if (email) conds.push(eq(users.email, email));
```

Before adopting A, confirm on the live harness that stored `users.email` is already lowercased on write (auth/signup path). If email case-normalization is **not** guaranteed at write time, keep `lower()` on email and take Option B for email only.

- [ ] **Option B (fallback): functional indexes**

Mirror the inquiries pattern (`inquiries_phone_lower_idx`/`inquiries_email_lower_idx`) by adding functional indexes to the users table, then keep the `lower()` predicate as-is:

```ts
// auth.ts users index block, alongside the existing uniqueIndexes
index("users_phone_lower_idx").on(sql`lower(${t.phone})`),
index("users_email_lower_idx").on(sql`lower(${t.email})`),
```

Then `pnpm --filter web db:generate` (folds into Task 1's migration if run together, or its own). `index` is already imported in `auth.ts`.

## Deferred / not worth it

- **`order` read on the `pickDish` not-found path (Task 4 Step 4)** — parallelizing `order`+`week` means `week` is fetched even when `order` is missing. This is one extra cheap indexed lookup on a rare error branch; not worth special-casing back to serial. Called out so the reviewer doesn't re-flag it.
- **Rewriting `phone-input.tsx` internals** (tree-shaking the flags module by country) — out of scope. The `next/dynamic` wrapper at the three dashboard consumers (Task 2) removes it from those routes' first paint, which is the whole win; restructuring the shared component for marginal extra savings isn't justified.
- **Server-side counts for `OrdersList` (the full `InquiriesList` parity path)** — Task 3's single-pass memo captures the per-keystroke win with a purely local change. Threading server-computed `statusCounts` as a prop (to match `InquiriesList` exactly) is a larger cross-file change for no additional client-perf benefit once the counts are memoized; defer unless a counts/total divergence (e.g. counting beyond the rendered page) is actually needed.
- **Option B of Task 5 by default** — adding two functional indexes to `users` is only warranted if email case-normalization at write time can't be guaranteed; otherwise Option A (raw equality, zero migration) is strictly simpler. Don't add the indexes speculatively while the users table is small.
