# Tiffin Grab — Dual-ID (snowflake bigint + prefixed nanoid) Design

**Date:** 2026-06-19
**Status:** Approved (brainstorming) — pending spec review
**Topic:** Replace UUID primary keys across the whole schema with a dual-id scheme: an internal snowflake `bigint` (relational key) plus a public prefixed nanoid (`public_id`) for all external surfaces.

## Goal

Every table carries two identifiers:

- **`id` — internal `bigint`**, generated by a Postgres snowflake `next_id()` function. This is the primary key and **the target of every foreign key**. Never crosses the wire.
- **`public_id` — `text`**, a 3-letter-prefixed 12-char nanoid (`usr_V1StGXR8z6Jd`), `NOT NULL UNIQUE`, app-generated. The **only** id exposed in URLs, API payloads, logs, and `session.user.id`.

This follows the PlanetScale/Instagram pattern (internal sequential key for storage/joins; opaque public id for the API), adapted to single-node Postgres.

## Context

- Monorepo (Turborepo + pnpm), Next.js 16 App Router, Drizzle/Postgres, Auth.js v5 (JWT strategy + DrizzleAdapter). Local dev Postgres at `postgres://lawbringr@localhost:5432/tiffin`.
- Ids are centralized: `packages/commons-drizzle/src/columns.ts` exports `baseColumns` / `updatableColumns`, currently `id: uuid("id").defaultRandom().primaryKey()`, `createdBy/updatedBy: uuid(...)`. Every table spreads these.
- ~18 FK columns across `apps/web/db/schema/{auth,orders,menu,inquiries,user-feature-flags}.ts` use `uuid(...).references(...)`.
- Services use `UpdatableRepository(db, table, table.id)`; routes/actions receive ids from clients and pass them to services. Auth ownership checks (e.g. `pickDish`) compare `session.user.id` to `order.userId`.
- **No production data.** The dev DB is reseedable (`db:seed`, `db:seed:catalog`, `db:seed:menu`, `db:seed:admin`). The migration baseline can be reset freely.

## Decisions (locked)

| Decision | Choice |
|---|---|
| ID style | Prefixed nanoid for `public_id`; prefixes **always exactly 3 lowercase letters** |
| nanoid length | **12 chars**, default url-safe alphabet (64 chars). `public_id` is public-only, so 12 is ample (64¹² ≈ 4.7×10²¹). |
| Internal id | **Snowflake `bigint`** via plpgsql `next_id()` (Instagram layout), generated **DB-side** |
| nanoid generation | **App-side** Drizzle `$defaultFn` |
| Foreign keys | Reference the **internal `bigint` `id`** |
| Auth.js | `users.id` = internal bigint; `users.public_id` = `usr_…`; jwt/session callbacks set `session.user.id = public_id` |
| Migration | **Reset baseline** — delete old migrations + snapshots, drop+recreate dev DB, reseed |
| Timestamps | **Epoch milliseconds** in `bigint` (Drizzle `mode:"number"`), app-side defaults. Auth-adapter timestamp columns carved out (stay as the adapter requires). |
| Scope shape | **One spec, one plan, 3 phases**, green at end (transient red mid-plan) |

## Snowflake `next_id()`

64-bit layout (Instagram): `41 bits ms-since-epoch | 13 bits shard | 10 bits sequence`.

- **Custom epoch:** `2025-01-01T00:00:00Z` (1735689600000 ms) → ~69-year lifespan on 41 bits.
- **Shard id:** default `1` (the future multi-shard knob; single node today).
- **Sequence:** a global Postgres sequence `id_seq`, taken `% 1024` per call (10 bits).
- Implemented as a `plpgsql` function plus the sequence, both authored as **raw SQL prepended to the baseline migration** (drizzle-kit does not manage function bodies; the baseline is hand-curated after generation).

Reference implementation (to be finalized in the plan):
```sql
CREATE SEQUENCE IF NOT EXISTS id_seq;

CREATE OR REPLACE FUNCTION next_id(OUT result bigint) AS $$
DECLARE
  our_epoch  bigint := 1735689600000;  -- 2025-01-01Z
  seq_id     bigint;
  now_millis bigint;
  shard_id   int := 1;
BEGIN
  SELECT nextval('id_seq') % 1024 INTO seq_id;
  SELECT floor(extract(epoch FROM clock_timestamp()) * 1000) INTO now_millis;
  result := (now_millis - our_epoch) << 23;  -- 13 shard + 10 seq = 23
  result := result | (shard_id << 10);
  result := result | seq_id;
END;
$$ LANGUAGE plpgsql;
```

Column: `bigint("id", { mode: "bigint" }).primaryKey().default(sql\`next_id()\`)`.

## The ID factory (`commons-drizzle`)

`baseColumns` / `updatableColumns` become **factories** taking the 3-letter prefix, because a shared static `id` cannot know each table's prefix:

```ts
updatableColumns("usr") // =>
{
  id:        bigint("id", { mode: "bigint" }).primaryKey().default(sql`next_id()`),
  publicId:  text("public_id").notNull().unique().$defaultFn(() => `usr_${nanoid(12)}`),
  createdAt: bigint("created_at", { mode: "number" }).notNull().$defaultFn(() => Date.now()),
  createdBy: bigint("created_by", { mode: "bigint" }),       // → users.id (nullable)
  updatedAt: bigint("updated_at", { mode: "number" }).notNull().$defaultFn(() => Date.now()).$onUpdate(() => Date.now()),
  updatedBy: bigint("updated_by", { mode: "bigint" }),       // → users.id (nullable)
}
```

- `nanoid` added as a dependency of `@tiffin/commons-drizzle`.
- Every table changes `{ ...updatableColumns, ... }` → `{ ...updatableColumns("xyz"), ... }`.
- `createdBy`/`updatedBy` are plain bigints (hold a user's internal id; do not self-generate). They reference `users.id` where a constraint is desired (matching current behavior — currently nullable, unconstrained audit columns; keep as-is unless the plan adds FKs).

## Prefix map (3-letter)

`usr` users · `acc` accounts · `ses` sessions · `flg` feature_flags · `uff` user_feature_flags · `pln` plans · `msz` meal_sizes · `frq` delivery_frequencies · `zon` delivery_zones · `dur` duration_packages · `ord` orders · `inq` inquiries · `iac` inquiry_activities · `slt` meal_slots · `dsh` dishes · `mnw` menu_weeks · `mni` menu_items · `msl` meal_selections.

(`accounts` and `verification_tokens` use composite PKs with no surrogate `id`; they get a `public_id` only if they spread the factory. Accounts/sessions are adapter-managed — see Auth.js. `verification_tokens` keeps its composite key, no public_id.)

## Risk A — 64-bit ids exceed JS safe integers

A snowflake value can exceed `2^53`, so JS `number` silently corrupts it. Rules:

- Drizzle `bigint` columns use **`mode: "bigint"`** (returns JS `BigInt`), never `mode: "number"`.
- The internal `id` is **never** serialized to the client, JSON, or JWT as a number. Only `public_id` (a string) crosses the wire.
- Any place that previously logged/returned `row.id` switches to `row.publicId` for external output; internal joins use the bigint.

## Risk B — Auth.js DrizzleAdapter

The adapter treats `users.id` as the account identifier. Approach:

- `users.id` stays internal bigint; `users.public_id` = `usr_…`.
- The DrizzleAdapter continues to manage `users`/`accounts`/`sessions` with bigint `userId` FKs internally (JWT strategy means the `sessions` table is largely unused at runtime, lowering risk).
- **jwt/session callbacks set `session.user.id = users.public_id`.** All application code (ownership checks, audit stamping, route params) keys off `public_id` at the boundary and resolves to the internal bigint for joins.
- Implementation must verify: adapter user creation/lookup works with a DB-defaulted bigint PK (adapter omits `id` on insert → DB generates → `RETURNING` provides it as BigInt), and credential login still resolves the user. Covered by an auth login smoke in Phase 2.

## Timestamps as epoch (ms)

All **domain** timestamp columns are stored as **epoch milliseconds** in `bigint` (Drizzle `mode:"number"` — ms epoch ≈1.7×10¹² is well under `2^53`, so JS `number` is lossless and no BigInt friction).

- Factory: `created_at`/`updated_at` → `bigint mode:"number"`, default `Date.now()`, `updated_at` also `$onUpdate(() => Date.now())`.
- Other domain timestamps converted from `timestamp(..)` to `bigint` ms: `menu_weeks.order_cutoff`, `menu_weeks.released_at`, inquiry timestamps, and any other `timestamp(...)` in non-auth schema. Application code reads/writes `number` (ms).
- **Epoch in / epoch out:** the DB, services, API, and loaders always store and return **raw UTC epoch ms** (`number`) — never a formatted or zone-shifted string. A single shared converter (`apps/web/lib/format/datetime.ts` — `formatEpoch(ms, {timeZone, mode})`, `epochToDate(ms)`) renders human-readable, **client-local** strings, and is called **only from client UI components** (using the client's own `Intl` timezone). No server component or service formats a timestamp.
- **Auth.js carve-out:** the DrizzleAdapter passes real `Date`/timestamptz for `sessions.expires`, `verification_tokens.expires`, `users.email_verified`; `accounts.expires_at` is OAuth epoch-**seconds** `integer`. These **stay as the adapter requires** — do not convert. Only our domain timestamps move to bigint ms.

## Boundary mapping (services/routes)

With FKs on bigint and clients passing `public_id`, the external↔internal seam is explicit:

- Services/repositories gain a `findByPublicId(publicId)` path; external operations (GET/PUT/DELETE by id, ownership checks) key on `public_id`.
- Internal joins and FK writes use the resolved bigint `id`.
- Routes/actions receive `public_id` (URL/body), resolve once, then operate internally.
- This is the deliberate, permanent cost of FKs-on-bigint; it is contained to Phase 3.

## Migration: reset baseline

1. Delete `apps/web/db/migrations/*.sql` and `apps/web/db/migrations/meta/*`.
2. Generate a fresh baseline from the new schema (`db:generate`), then **prepend** the `id_seq` + `next_id()` raw SQL to the baseline `.sql` (and keep it reflected in the hand-curated baseline; the snapshot tracks column defaults, not the function body).
3. Drop + recreate the dev database; `db:migrate`.
4. Reseed all four seeds; confirm idempotency + final counts.
